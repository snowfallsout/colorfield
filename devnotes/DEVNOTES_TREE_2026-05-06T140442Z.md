# DEVNOTES TREE 2026-05-06T140442Z

## Task 1: Baseline scan

### Plan
- Record the current ownership map from the existing repo structure.
- Identify the coupling points that block clearer modular ownership.
- Keep this step read-only so the design stays grounded in the current codebase.

### Todo
- Treat `routes/` as composition entry points only.
- Treat `components/` as presentation owners only.
- Treat `state/` as reactive data ownership only.
- Treat `services/` as imperative I/O and workflow owners.
- Treat `shared/` as cross-domain contracts and constants only.
- Treat `server/` as server-only persistence and socket adapters.

### Result
- The current project already has the right top-level layers.
- The weak spots are not missing folders, but blurred responsibilities:
  - display runtime still owns DOM lifecycle and socket binding together.
  - legacy bridge code still mirrors DOM ids and `window` globals.
  - some state modules still carry side effects and browser-only work.
  - `shared/contracts.ts` is doing too much at once for multiple domains.

### Impact
- The architecture can be improved without a full rewrite.
- The safest path is to tighten ownership boundaries and move side effects outward.

## Task 3: Step-by-step migration plan

### Plan
- Use an incremental migration so each step is reviewable and reversible.
- Move one responsibility at a time and validate immediately after each move.
- Avoid touching `static/` and avoid broad path churn.

### Todo
1. Freeze cross-domain contracts.
   - Split `shared/contracts.ts` into smaller contract files by responsibility:
     - session API contracts
     - socket event contracts
     - display payload contracts
   - Keep `shared/constants/` as the only source of shared MBTI and vision constants.
   - Replace any route or service imports that reach through unrelated contract groups.
2. Make `state/` pure.
   - Keep only reactive data and tiny mutation helpers in `.svelte.ts` files.
   - Remove direct DOM ownership from state modules.
   - Move browser-only work into service modules or route/component lifecycle hooks.
3. Make display services explicit.
   - Keep `runtime.ts` as the orchestration entrypoint only.
   - Keep `camera.ts`, `realtime.ts`, and `session.ts` as concrete effect owners.
   - Keep `draw.ts`, `gesture.ts`, `particle.ts`, and `sprite.ts` as render/runtime primitives.
   - Reduce `legacy.ts` to a temporary compatibility bridge.
4. Make routes thin.
   - `display/+page.svelte` should only compose display components and start the page-level orchestration.
   - `mobile/+page.svelte` should only mount the mobile shell.
   - No route should directly own socket, camera, or filesystem logic.
5. Remove legacy coupling.
   - Replace DOM id lookups where component state or props can express the same relationship.
   - Replace `window` globals with explicit module imports or typed adapters.
   - Keep any remaining legacy hooks isolated behind a compatibility facade.
6. Validate after each boundary move.
   - Run `npm run check`.
   - Run targeted runtime smoke checks for display and mobile.
   - Confirm no new `document` or `window` usage leaks into state-only modules.

### Result
- The project can evolve as a set of focused modules instead of one broad runtime surface.
- Each move has a clear owner, so regressions become easier to localize.

### Impact
- Smaller review surface per change.
- Lower risk of cross-domain regressions.
- Better long-term maintainability for both the live display flow and the mobile flow.
