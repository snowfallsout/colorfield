# DEVNOTES_2026-05-02T220000Z

## 背景

在前幾輪局部重構後，`src/lib` 仍殘留幾塊明顯重複內容：

- `state/display.svelte.ts` 內手寫 16 組 legend row 初始資料
- `utils/faceHash.ts` 重複定義 `SMILE_EMOJIS`
- `utils/mbti.ts` 重複定義 `MBTI_COLORS` 與 `MBTI_NAMES`

這些內容都已有或應有 shared owner，繼續保留副本只會讓局部修正互相漂移。

## 本輪決策

### A. display legend rows 改由 shared MBTI constants 生成

`state/display.svelte.ts` 不再手寫 16 筆 MBTI row，而是從：

- `MBTI_ORDER`
- `MBTI_COLORS`

動態生成預設 legend rows。

### B. `utils/faceHash.ts` 只保留 face-hash 與 random-picker 行為

`SMILE_EMOJIS` 改由 `shared/constants/vision.ts` 提供，避免 smile emoji 清單在多處維護。

### C. `utils/mbti.ts` 保留 mobile card 色彩工具，但不再重複 MBTI 基礎常數

`MBTI_COLORS`、`MBTI_NAMES` 改為直接 re-export shared owner，維持既有 import surface，避免一口氣擴散 consumer 變更。

### D. `displayState` 先不強行改成 rune-only API

目前 `services/display/session.ts` 等外部仍使用 `get(displayState)` 與 `subscribe` 介面。
因此這輪只移除無外部使用的 `derived(...)` 輸出，不直接重寫 `displayState` 公開 API。

## 驗證方式

本輪完成後需確認：

- `npm run check` 通過
- `state/*` 中 `svelte/store` 只剩真正仍被外部 `get(...)` / `subscribe` 依賴的 display state owner
- `SMILE_EMOJIS`、`MBTI_COLORS`、`MBTI_NAMES` 不再在 utils 層重複定義
