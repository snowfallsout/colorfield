# Protected Files Policy

/*
  File purpose:
  - Human-readable policy describing which repository paths are protected
    and the process for requesting changes.
*/

## Protected paths
- `static/**` — Static build artifacts and legacy HTML pages.
- `.github/**` — Repository agent instructions and workflows.

## Rationale
Files under `static/` and `.github/` are considered sensitive in this project:
- `static/` contains legacy pages and runtime assets that are not intended for live refactors without maintainer approval.
- `.github/` contains agent and CI instructions; accidental edits can change automation behaviour.

## Required process for changing protected files
1. Open an Issue describing the proposed change and rationale, tagging maintainers.
2. Wait for explicit approval from a maintainer in the issue (comment or review).
3. Create a DEVNOTES file in `devnotes/DEVNOTES_{ISO_TIMESTAMP}.md` describing the implementation plan and verification steps.
4. Submit a PR that includes the DEVNOTES and references the approval issue number. CI will run; maintainers review and merge.

## Agent behaviour (enforced)
- Agents must consult `/memories/repo/protected_files.json` before applying any patch.
- If a proposed patch touches a protected path, the agent must abort the patch and prompt the user to follow the process above.

## Exceptions
- Emergency fixes require a maintainer to mark the issue with `urgent:approved` and explicitly permit the change in the issue thread.
