# DEVNOTES TREE 2026-05-06T141431Z

## Task 1: Define classification rules

### Plan
- Use three buckets so the next refactor stays simple and reversible.
- Classify each file by ownership, active usage, and side effects.
- Prefer moving behavior to the real owner instead of widening `utils/`.

### Todo
- `Keep`: active, pure helper files with clear cross-domain reuse.
- `Move`: active files that already have a better home in `state/`, `services/`, `shared/`, or `server/`.
- `Retire`: legacy wiring, dead helpers, or compatibility layers that should disappear after callers move.

### Result
- The `utils/` folder is not one category.
- It currently mixes reusable helpers, domain helpers, and old runtime glue.

### Impact
- The plan avoids a blind mass split.
- Each file gets one explicit destination instead of a vague “later cleanup”.

## Task 3: Migration order

### Plan
- Remove the highest-confidence legacy files first.
- Keep active helpers until all callers are moved.
- Validate after each removal batch so regressions stay localized.

### Todo
1. Freeze the current callers.
   - Confirm `state.ts`, `mbti.ts`, `ml-camera.ts`, `coordinator.ts`, `overlay.ts`, `emotion.ts`, `drawmode.ts`, `particle.ts`, `pool.ts`, and `holoCanvas.ts` are not used by the new source tree.
   - Keep `session.ts`, `sprites.ts`, `faceHash.ts`, and `types.ts` under review because they still have active or structural roles.
2. Move surviving active helpers.
   - If `session.ts` remains active, keep it small and browser-safe.
   - If `types.ts` remains active, move it into a shared or domain type module.
   - If `sprites.ts` remains active, move it under the display service only when the display owner is ready to own it fully.
   - If `faceHash.ts` remains active, move it to the closest shared display helper location.
3. Delete the retired legacy files in one controlled batch.
   - Remove only the files that have zero active references and no compatibility requirement.
   - Do not delete `state.ts` until the display-state compatibility surface is confirmed safe.
4. Validate the cleanup.
   - Run `npm run check` after the batch.
   - Re-scan imports for any remaining `$lib/utils/*` references.
   - Update `app.d.ts` if any fake module declarations become stale.

### Result
- The folder shrinks in a controlled way instead of being split randomly.
- The remaining files are easier to reason about because their roles are explicit.

### Impact
- Lower risk of breaking display/mobile runtime paths.
- Cleaner ownership map for the next implementation pass.
