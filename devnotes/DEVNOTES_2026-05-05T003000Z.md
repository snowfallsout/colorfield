# DEVNOTES_2026-05-05T003000Z

## 背景

display route 目前有兩個問題：

- 進入頁面就跳 browser camera permission，而不是像 `static/display.html` 一樣由使用者手動開啟
- `水幕 ON | OFF` 目前只切 CSS overlay，沒有真正影響 canvas 背景填色

進一步檢查後，原因是目前同時存在兩條 camera owner：

- 新路徑：`state/media.svelte.ts` -> `services/mediapipe.ts`
- 舊路徑：`components/display/Canvas.svelte` -> `utils/coordinator.ts` -> `utils/ml-camera.ts`

舊路徑會在 mount 時直接 `getUserMedia()`，因此會搶先跳權限視窗，也會和新 camera toggle 路徑互相衝突。

## 本輪決策

### A. 移除 Canvas 上的舊 camera owner

`Canvas.svelte` 不再 mount `utils/coordinator.ts`。display route 的 camera 只保留：

- `CamToggle.svelte`
- `state/media.svelte.ts`
- `services/mediapipe.ts`

### B. 增加 camera loading state

`media.svelte.ts` 新增 `camLoading`，讓 UI 可以在 script/model/camera stream 載入期間顯示 loading 動畫並暫時禁用按鈕。

### C. 水幕 toggle 改為控制 canvas 背景填色模式

`ui.waterOverlay` 現在除了 CSS overlay，也會控制 `Canvas.svelte` 的背景 fill alpha：

- OFF: 不透明背景，回到穩定白底
- ON: 半透明背景，保留較強的水幕/拖尾感

這個開關不再和 camera state 綁定。

## 驗證方式

- 進入 `/display` 不應自動要求 camera permission
- 點擊 camera 按鈕後才會請求 camera permission
- camera loading 期間應有明確的 loading UI
- `水幕 ON | OFF` 應獨立影響 canvas 背景 fill，而不是等同 camera switch
- `npm run check` 通過
